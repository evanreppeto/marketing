# Agent Contract v1

Contract version: v1. Additions within v1 must be backwards-compatible.

This document defines the HTTP contract for plugging a Arc-compatible agent into the Growth Engine app. The app is the control plane: it owns records, queues, approvals, audit logs, and outbound locks. The agent reads work, produces drafts or recommendations, and reports progress.

## Safety Boundary

Agents may read, draft, recommend, log, and manage their own task lifecycle.

Agents must not approve, launch, send, publish, dispatch, or unlock outbound work. Public-facing work must land in app-owned approval state first.

## Auth

Agent to app calls use:

```http
Authorization: Bearer <token>
```

The token can be either:

- `ARC_AGENT_API_TOKEN` from the app environment.
- An app-issued `sk_live_...` token generated in Settings -> Agent.

App-issued tokens are stored only as SHA-256 hashes. Plaintext is shown once.

## Wake: App To Agent

When an operator sends a Arc chat message, the app can wake the configured agent webhook.

```http
POST <webhookUrl>
Content-Type: application/json
X-Webhook-Signature: <hmac-sha256-hex>
```

`X-Webhook-Signature` is omitted when no signing secret is configured. When present, it is the HMAC-SHA256 hex digest of the raw JSON body using the shared signing secret.

Body:

```json
{
  "type": "arc_chat_message",
  "messageId": "uuid",
  "conversationId": "uuid",
  "agentTaskId": "uuid",
  "message": "Find partner leads for next week.",
  "mentions": [
    { "type": "campaign", "id": "uuid", "label": "Partner reactivation", "href": "/campaigns/uuid" }
  ],
  "operator": "operator@example.com",
  "route": "fast",
  "mode": "ask",
  "command": "find-leads",
  "attachments": []
}
```

The webhook should acknowledge quickly with any 2xx response and do slower work asynchronously. If the wake is missed, the agent can pull queued messages from the inbox.

## Health

```http
GET /api/v1/arc/ping
Authorization: Bearer <token>
```

Returns connectivity and persistence readiness:

```json
{
  "ok": true,
  "status": "connected",
  "service": "arc",
  "supabaseConfigured": true,
  "time": "2026-06-11T00:00:00.000Z"
}
```

```http
GET /api/v1/arc/health
Authorization: Bearer <token>
```

Returns service health without requiring Supabase to be configured.

## Chat Inbox And Replies

Pull queued chat work:

```http
GET /api/v1/arc/messages?limit=20
Authorization: Bearer <token>
```

Response:

```json
{
  "ok": true,
  "status": "ok",
  "messages": [
    {
      "agentTaskId": "uuid",
      "conversationId": "uuid",
      "message": "What should we draft?",
      "mentions": [],
      "operator": "operator@example.com",
      "createdAt": "2026-06-11T00:00:00.000Z"
    }
  ]
}
```

Deliver a reply:

```http
POST /api/v1/arc/messages
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "agentTaskId": "uuid",
  "body": "Here is the recommended plan.",
  "status": "complete",
  "metadata": {
    "actions": [],
    "media": []
  }
}
```

Use `status: "failed"` with a short body when the agent cannot complete the request.

Report live steps:

```http
POST /api/v1/arc/messages/{agentTaskId}/steps
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "steps": [
    { "label": "Reading campaign context", "status": "complete" },
    { "label": "Drafting response", "status": "active" }
  ],
  "body": "Working..."
}
```

Steps are best-effort UI telemetry. The final reply still settles through `POST /api/v1/arc/messages`.

## Tasks

List tasks:

```http
GET /api/v1/arc/tasks?status=queued&assignee=arc&limit=20
Authorization: Bearer <token>
```

Get task detail:

```http
GET /api/v1/arc/tasks/{id}
Authorization: Bearer <token>
```

Claim a task:

```http
POST /api/v1/arc/tasks/{id}/claim
Authorization: Bearer <token>
```

Append a run log:

```http
POST /api/v1/arc/tasks/{id}/log
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "message": "Loaded campaign and persona context.",
  "reasoning_summary": "Partner reactivation is relevant because the account has been dormant.",
  "run_status": "running",
  "model_provider": "custom",
  "model_name": "arc",
  "metadata": {}
}
```

Complete a task:

```http
POST /api/v1/arc/tasks/{id}/complete
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "summary": "Drafted a reviewable partner outreach plan.",
  "outputs": [],
  "metadata": {}
}
```

Block a task:

```http
POST /api/v1/arc/tasks/{id}/block
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "reason": "Missing campaign context.",
  "needs": ["campaign_id"],
  "metadata": {}
}
```

## Drafts And Approvals

Create a human-reviewable draft:

```http
POST /api/v1/arc/drafts
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "item_type": "social_ad",
  "title": "Partner water backup ad",
  "summary": "Drafted for owner review.",
  "draft": { "body": "Draft copy..." },
  "risk_level": "low",
  "prompt_inputs": {},
  "campaign_id": "uuid",
  "task_id": "uuid",
  "metadata": {}
}
```

The app creates approval-locked state. The agent does not approve it.

List approvals:

```http
GET /api/v1/arc/approvals?status=pending_owner_approval&limit=20
Authorization: Bearer <token>
```

Get an approval:

```http
GET /api/v1/arc/approvals/{id}
Authorization: Bearer <token>
```

Add an advisory recommendation:

```http
POST /api/v1/arc/approvals/{id}/recommendation
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "recommendation": "approve",
  "rationale": "The copy stays coverage-neutral and matches the approved brief.",
  "risk_flags": [],
  "suggested_edits": [],
  "agent": "arc",
  "metadata": {}
}
```

Read recommendations:

```http
GET /api/v1/arc/approvals/{id}/recommendations
Authorization: Bearer <token>
```

## Fallback Behavior

If the app cannot wake the webhook, it leaves the task queued. The agent should poll `GET /api/v1/arc/messages` or `GET /api/v1/arc/tasks` to catch up.

If the agent starts work but cannot finish, it should either:

- Post a failed chat reply with `status: "failed"`.
- Block the task with `/tasks/{id}/block`.
- Append logs with `/tasks/{id}/log` so the operator can inspect what happened.

## Compatibility Rules

- Unknown response fields should be ignored.
- New request fields may be added as optional fields.
- Existing required fields will not be removed within v1.
- Agents should treat public outbound actions as blocked unless a future contract explicitly grants a scoped execution capability.
